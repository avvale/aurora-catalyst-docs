## When to use

You have just archived (or plan to reflect) a feature in `aurora-catalyst-cli` (or another Aurora repo) and need to update the docs site. The archive lives at `<cli-repo>/openspec/changes/archive/YYYY-MM-DD-<change-name>/`.

You do NOT use this skill for:

- Auto-generated reference pages (CLI commands, TypeDoc API, change history mirror). Those are regenerated by `scripts/import-from-catalyst.ts` ‚Äî running the script is enough.
- Tutorials. These are curated by a human; the skill only suggests updates, it does not write a tutorial from scratch.

## Required context before you start

1. **Path of the source repo** where the archive lives (defaults to `../aurora-catalyst-cli` relative to this repo). Confirm with the user or use `--add-dir` / `additionalWorkingDirectories`.
2. **Change identifier** ‚Äî the archive directory name (e.g. `2026-04-18-preservation-regions`).
3. The canonical spec synced to `<cli-repo>/openspec/specs/<capability>/spec.md` if it exists.

## Inputs you MUST read

In this order:

1. `<cli-repo>/openspec/changes/archive/<change>/proposal.md` ‚Äî the "why" and the scope (`Impact` section lists the files the change touched).
2. `<cli-repo>/openspec/changes/archive/<change>/design.md` ‚Äî architectural decisions and the rationale behind each.
3. `<cli-repo>/openspec/changes/archive/<change>/specs/<capability>/spec.md` OR the canonical spec at `<cli-repo>/openspec/specs/<capability>/spec.md` ‚Äî the normative requirements and scenarios.
4. The **source files mentioned in Impact** ‚Äî enough to confirm the public surface (exported functions, new flags, types). Do not read more than you need.

Do NOT read the tasks.md file in detail. It is implementation history; the design doc is a better source of "why".

## What you produce

Decide per-change which categories apply. Not every change needs all four.

### 1. Concept page (almost always)

Create or update `src/content/docs/en/concepts/<topic>.md` AND `src/content/docs/es/concepts/<topic>.md`.

Purpose: explain the _why_ of the feature. Reader already knows Aurora exists; they want to build a mental model.

Template skeleton:

```markdown
---
title: <Topic>
description: <one-line description>
---

## Why this exists

<1-2 paragraphs. Lift from proposal.md "Why" section but rewrite conversationally. Name the problem. Name the user pain.>

## How it works

<2-4 paragraphs or a diagram. Based on design.md decisions. Focus on the mental model, not exhaustive detail.>

## When it applies

<Bullet list of scenarios where the reader encounters this concept. E.g.: "You see `[REGION PRESERVED]` in the output", "Your lockfile grows a `regions` field".>

## Trade-offs and limits

<From design.md "Risks / Trade-offs" and Non-Goals. Be honest. Readers trust docs that name limits.>

## Related

- [How-to: ‚Ä¶](/guides/‚Ä¶/)  ‚Üê link to the practical recipes
- [Reference: ‚Ä¶](/reference/‚Ä¶/)  ‚Üê link to auto-generated parts
```

### 2. How-to guide (when the feature changes user workflow)

Create or update `src/content/docs/en/guides/<area>/<action>.md` AND the Spanish mirror at `src/content/docs/es/guides/<area>/<action>.md`.

Purpose: solve a concrete task. Reader has a goal.

Template skeleton:

```markdown
---
title: <Action-oriented title, e.g. "Add a preservation region to a form template">
description: <one-line>
---

## Goal

<One sentence: what the reader will achieve by following this guide.>

## Before you start

<Prerequisites: what they must already have set up.>

## Steps

1. **<Verb-first step>.** <Code block or concrete action.>
2. ...
3. ...

## Verify it worked

<How to confirm success. CLI output, file state, test command.>

## Troubleshooting

<1-3 common pitfalls with their fix.>

## Related

- [Concept: ‚Ä¶](/concepts/‚Ä¶/)
```

Guides are task-focused. Do NOT explain _why_ here ‚Äî link to the concept page.

### 3. Reference stub (when the feature adds a new structural surface)

For new YAML fields, new lockfile fields, new file conventions: hand-write a stub in `src/content/docs/en/reference/<topic>.md` (+ Spanish mirror). For new CLI commands or API: do nothing ‚Äî those come from `pnpm import`.

Template skeleton:

```markdown
---
title: <Topic>
description: <one-line>
---

## Schema

<Exact structure. Table or code block.>

## Fields

- `<fieldName>` (type, optional/required): <description>
- ...

## Example

<Minimal working example. Copy-paste-ready.>
```

### 4. Tutorial update (rare)

Only when the feature changes the getting-started experience or introduces a step a new user must know from day one. Do NOT write new tutorials ‚Äî raise it with the user and let them decide.

## Language coverage

Every page you write MUST exist in both EN (`src/content/docs/en/‚Ä¶`) and ES (`src/content/docs/es/‚Ä¶`) at symmetric paths. Write the EN version first, then the ES version ‚Äî do NOT machine-translate, rewrite idiomatically in Rioplatense Spanish (voseo) following the user's preference.

Both versions live in the same PR.

## Quality bar ‚Äî before you open the PR

- [ ] Each page has a `title` and `description` in the frontmatter.
- [ ] No duplicated prose between concept / guide / reference. Cross-link instead.
- [ ] Links use the Starlight absolute form WITH the locale prefix (`/en/concepts/foo/`, `/es/concepts/foo/`), not relative paths like `../concepts/foo.md`.
- [ ] No broken image references, no placeholder `TODO` left without an explicit `:::note[Placeholder]` block.
- [ ] Spanish page is NOT a literal translation ‚Äî paragraphs are idiomatic and may reorder.

## What you do NOT do

- Do NOT modify `scripts/import-from-catalyst.ts`. That is plumbing for auto-generated pages.
- Do NOT touch anything under `{en,es}/reference/cli-commands/`, `{en,es}/reference/api/`, or `{en,es}/changes/` ‚Äî those are overwritten by the import script.
- Do NOT create categories outside the Di√°taxis four (tutorials, guides, reference, concepts). If something doesn't fit, raise it with the user.
- Do NOT delete or rename existing files without explicit user confirmation. Docs URLs are contracts.

## Hand-off

When done:

1. Summarize in a PR description which Di√°taxis categories you touched and why.
2. Link the PR to the source archive: `Documents change: <cli-repo>/openspec/changes/archive/<change>/`.
3. Run `pnpm build` locally to catch broken links before pushing. If you can't, say so explicitly.
