## Why

The current mirror of the openspec archive under `src/content/docs/{en,es}/changes/` replicates full proposal + design + tasks + spec files (hundreds of lines per change) that are not useful to a developer who wants to see what has actually changed in the platform. We want a genuinely useful changelog section: scannable, user-facing, bilingual, and generated by reusing the real archive as input — without manual rewriting and without publishing internal noise (refactors, performance tweaks, bug fixes).

## What Changes

- **BREAKING** Remove the current archive mirror at `src/content/docs/{en,es}/changes/cli/<slug>/` (proposal.md, design.md, tasks.md, index.md, specs/*). URLs under `/changes/cli/<slug>/` for existing mirrored changes will be replaced by a new shape. Requires explicit user confirmation before deletion.
- Introduce a deterministic sync script that, for each configured source repo, scans its `openspec/changes/archive/` and diffs against a versioned `.processed.json` registry kept in this repo.
- Introduce a skill (invoked by the developer) that runs the script and, for every unprocessed change, asks the LLM to classify it as **PUBLISH** or **DISMISS** using a fixed, user-facing-impact contract (see design).
- For **PUBLISH** changes, the LLM drafts an entry in English AND Spanish (tuteo neutral) derived from the source archive's `proposal.md` (narrative) and `specs/<capability>/spec.md` (final behaviour). No other archive files are read.
- For **DISMISS** changes, only a registry entry with a one-line reason is written; no docs pages are generated.
- Persist every decision (published + dismissed) in `.processed.json`, versioned in the repo, keyed by `<repo>/<slug>`. `source_commit` is stored for auditability; `classification` is stored only for published entries.
- Provide manual override (bidirectional): the skill accepts an instruction to force-publish a previously dismissed change or to force-dismiss a previously published one. Overrides are re-run through the same LLM pipeline and recorded.
- Group the index `/changes/cli/` by semver tag read from the source repo (`git tag --contains <archive_commit> --sort=v:refname | head -1`), with an `Unreleased` bucket at the top for archived changes not yet contained in any tag. Re-runs reshuffle automatically when new tags appear.
- Generalise the design so a second source repo (e.g. `aurora-catalyst` framework) can be added without reworking the pipeline.
- CLI commands sync (`reference/cli-commands/`) remains deterministic and out of scope — explicitly untouched.

## Capabilities

- `changelog-sync`: detect new archived changes in configured source repos, classify them against a user-facing-impact contract, author bilingual entries for the ones that qualify, maintain a versioned decision registry, and render a semver-grouped index with per-change detail pages.

## Impact

- **Removed docs content**: `src/content/docs/en/changes/cli/2026-04-18-preservation-regions/**` and its Spanish counterpart (currently the only mirrored change), plus the related entries in `changes/index.md` and `changes/cli/index.md`.
- **New repo artefacts**:
  - `.processed.json` (path to be finalised in design) — versioned registry of published + dismissed decisions.
  - New skill under `.claude/skills/` for the sync + classify + author flow.
  - New or extended script under `scripts/` (alongside the existing import script) to compute the delta and emit the LLM input payload.
  - New page templates under `src/content/docs/{en,es}/changes/` for the index and per-change entry shape.
- **Cross-repo contract**: this repo now depends on source repos exposing an `openspec/changes/archive/` directory and (optionally) semver git tags. It continues to never write to sibling repos.
- **Existing `pnpm sync` pipeline**: CLI commands and API reference imports are unchanged. The changelog step becomes skill-driven rather than script-only because it requires LLM reasoning; the rest of `pnpm sync` stays deterministic.
- **CI / GitHub Pages deploy**: unaffected. CI keeps trusting whatever is committed. The skill runs locally before commit, same as today's `pnpm sync`.
